imgthing.

A single-owner photo library that lives entirely on Cloudflare — one Worker, three bindings, and no other vendor. This is how it fits together, and how to run your own.

The idea

One vendor, one owner, one Worker

imgthing is a personal photo library for exactly one person. That single decision removes almost all the usual complexity — there is no user table, no per-row ownership scoping, no cross-tenant isolation. Everything runs inside one Cloudflare Worker that serves both the app and its API, backed by Cloudflare's own storage primitives.

Originals live in R2. Metadata lives in D1 (SQLite). Resizing happens once at upload through the Cloudflare Images binding, and every image view after that is a plain read from R2. Nothing depends on a custom domain — the whole thing works on a *.workers.dev URL from the first deploy.

The stack at a glance

Five moving parts

Compute

Nuxt 4 on Workers

SSR via the Nitro cloudflare_module preset. Serves the UI and every API route.

Metadata

Cloudflare D1

SQLite at the edge. Photos, folders, EXIF, tags — nowhere near D1's limits at one-owner scale.

Originals

R2 bucket

One object per photo, plus three precomputed variants. Free egress, private bucket.

Resize

Images binding

env.IMAGES as an upload-time resize function only — never on a serving path.

Gate

Hand-rolled auth

One passphrase, a signed cookie. Self-contained and testable with no Cloudflare account.

Request flows

What happens on each path

Signing in

1

Submit passphrase

POST /api/auth/login compares it to APP_PASSPHRASE in constant time.

2

Mint a cookie

An HMAC-SHA256 cookie imgthing_session signed with SESSION_SECRET. Payload {sub, exp}, 30 days, httpOnly.

3

Two guards

Server middleware 401s any /api/**; page middleware redirects to /login. No sessions table.

Uploading a photo

1

Store the original

Bytes go straight to R2 (one object). EXIF is parsed and written to D1.

2

Precompute variants

The IMAGES binding makes three WebP sizes — thumb 800, md 1280, lg 2560 — written to variants/{id}/{size}.

3

Non-fatal by design

Row + original are already durable. A failed resize just leaves the variant unmade — the first view heals it.

Viewing (owner)

1

Gated request

GET /api/photos/:id/variant?size= passes the auth middleware.

2

Read, don't transform

getOrCreateVariant streams the WebP from R2 on a hit; on a miss it regenerates all sizes, then serves.

3

Long cache

Served private, immutable for a year — the bytes never change.

Sharing (public)

1

Publish mints a token

A fresh 128-bit random public_token. Re-publishing rotates it — old links die permanently.

2

Tokened read

GET /p/:token/:size lives outside /api, so no session is needed. It checks token + visibility='public' + not trashed.

3

Uniform 404s

Bad size, bad token, private, or trashed all return an identical bare 404 — a revoked link can't be told from a fake one. Location EXIF is stripped by the WebP re-encode.

Storage model

Where the bytes live

Two stores, cleanly split: R2 holds the pixels, D1 holds everything you can query. A photo id ties them together.

R2  ·  BUCKET

Private bucket, never directly reachable. Keys:

<r2_key>                original bytes
variants/{id}/thumb     WebP · 800px
variants/{id}/md        WebP · 1280px
variants/{id}/lg        WebP · 2560px

D1  ·  DB

Eight migrations, 0001–0008. Core tables:

photos      metadata, r2_key, visibility,
            public_token, deleted_at
exif_data   camera / GPS, per photo
folders     + folder_photos join
tags        + favorites

Soft delete and unpublish are the same instant: trashing a photo clears visibility, public_token, and published_at in the update that sets deleted_at. Hard delete also removes the three variant objects and the original from R2.

The four bindings

Everything the Worker can touch is declared in wrangler.jsonc. Names matter — the code reads them directly.

Two secrets are set separately, not in wrangler.jsonc: APP_PASSPHRASE (your login) and SESSION_SECRET (a long random cookie-signing key). Locally they live in .dev.vars; in prod, wrangler secret put.

Run your own

Host it yourself

You need a Cloudflare account with Workers Paid ($5/mo, for CPU headroom) and Node locally. Everything below is account-scoped, so it's yours to run once.

  1. Authenticate & provision

    Log in, then create the two stateful stores.

    npx wrangler login
npx wrangler d1 create imgthing
npx wrangler r2 bucket create imgthing-photos

  2. Wire the database id

    Paste the printed database_id into the d1_databases block in wrangler.jsonc. Keep binding: "DB" and migrations_dir: "server/db/migrations" — wrangler d1 create tends to rewrite both.

  3. Apply migrations

    Runs every pending migration (0001–0008) against the remote D1.

    npm run db:migrate:remote

  4. Enable Cloudflare Images

    In the dashboard: account → Images. This is the account-level toggle that unlocks the IMAGES binding (free tier: 5,000 transforms/month). It is not the per-zone URL-transform setting, and it needs no custom domain.

  5. Set the two secrets

    Generate a strong random key without depending on a system openssl.

    npx wrangler secret put APP_PASSPHRASE
node -e "console.log(require('crypto').randomBytes(32).toString('hex'))" \
  | npx wrangler secret put SESSION_SECRET

  6. Deploy

    Builds Nuxt and publishes the Worker to a *.workers.dev URL. First visit lands on /login.

    npm run deploy

  7. (Optional) Custom domain

    Add your zone to Cloudflare, then attach it under Workers → your worker → Domains & Routes. Works on the free DNS plan. Variants, sharing, and everything else behave identically to the workers.dev URL.

Why it's built this way

Four decisions worth knowing

ADR 0005

Precomputed variants, not on-the-fly

An earlier design resized per-request through /cdn-cgi/image/ URLs — which forced a custom domain, a purge token, and per-view billing. Precomputing three WebP sizes at upload and serving them as plain R2 reads removes all of it. Steady-state serving cost is R2 reads only.

ADR 0003

Hand-rolled auth over Access

Cloudflare Access is excellent but couples auth to the Zero Trust dashboard and doesn't run in local dev without stubbing. One passphrase plus a signed cookie keeps the app self-contained, portable, and testable with no Cloudflare account.

ADR 0005

Uniform 404 on every public miss

Bad token, private, or trashed all return an identical bare 404. Without it, Nuxt would render the SPA shell at HTTP 200 for a non-/api route — making a revoked link look alive and leaking app HTML.

ADR 0001

Single-tenant, on purpose

No user table, no per-row scoping, one vendor. Portability is reduced by design in exchange for near-zero cost and a dramatically smaller surface. Multi-user would be a real rewrite — and that's fine.
